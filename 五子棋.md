# 五子棋程式規格：棋型、禁手與 VCF

> 本文件的目的，是讓沒有五子棋／連珠背景的 AI 或程式設計者，只靠這份文件就能實作出可驗證、可維護的棋型判斷、黑棋禁手判斷與 VCF 搜尋。
>
> 本文件同時區分：
>
> - **一般五子棋（Gomoku）**：黑白通常都沒有三三、四四、長連禁手。
> - **連珠（Renju）**：只有黑棋受到三三、四四、長連限制；白棋沒有這些限制。
>
> 若專案採用其他規則，必須把勝負與禁手規則做成明確參數，不可混在棋型判斷內。

---

## 1. 基本資料與符號

### 1.1 棋盤

標準連珠棋盤是 15×15，共 225 個交叉點。

座標：

```text
x = 0..14，從左到右
y = 0..14，從上到下
```

若使用一維陣列：

```text
idx = y * 15 + x
x = idx % 15
y = idx / 15
```

四個方向：

```text
0：橫線  —
1：直線  │
2：斜線  ↘
3：斜線  ↗
```

每個方向都必須有正、反兩側。棋盤外必須視為阻斷點，不可把相鄰列或相鄰斜線接在一起。

### 1.2 本文件棋型符號

```text
X：目前分析方的棋子
O：對手棋子
.：空點
#：棋盤外
@：本次新落下、正在判斷的棋子
```

範例：

```text
. X X @ . .
```

表示同一條線上有三顆己方棋，其中 `@` 是本次落子。

### 1.3 棋盤狀態的唯一真實來源

可使用：

```text
byte board[225]
```

或 Bitboard：

```text
blackBits
whiteBits
```

也可以兩者並存，但必須明確指定哪一份是唯一真實狀態，另一份只能是同步快取。每次落子與回溯都必須同步且可驗證。

## 2. 規則層與棋型層必須分開

程式至少應拆成三層：

```text
棋盤層：放子、取子、讀格、方向索引
棋型層：某一條線形成了什麼棋型
規則層：這一步是否合法、是否禁手、是否獲勝
```

棋型層只回答「形狀與威脅是什麼」；規則層才回答「黑棋能不能下」。

錯誤做法：

```text
看到形狀像三，就立刻判三三。
```

正確做法：

```text
先找出表面上可能成立的三，
再驗證它是否存在合法的活四延伸點，
只有驗證成功才是真正的三。
```

## 3. 官方概念與程式化定義

### 3.1 同線棋列（Row）

同色棋子位於同一條橫、直或斜線上，中間不能有對手棋子。

空點可以存在，因此：

```text
X X . X
```

仍可視為同一條同色棋列的一部分，但不是連續棋列。

### 3.2 連續棋列（Unbroken Row）

棋子之間沒有空點：

```text
X X X X
```

### 3.3 正五（Exact Five）

同一方向上恰好連續五顆：

```text
. X X X X X .
```

對黑棋連珠規則而言，必須是「恰好五顆」，不能是六顆以上的一部分。

### 3.4 長連（Overline）

同一方向上連續六顆或更多：

```text
X X X X X X
```

連珠規則：

```text
黑棋長連：禁手
白棋長連：可作為勝利
```

一般五子棋是否把長連視為勝利，應由規則參數決定。

### 3.5 成五點（Winning Point）

對某一方而言，若在空點 `p` 落子後能完成該規則允許的勝利棋列，則 `p` 是成五點。

程式介面可定義為：

```text
winningPoints(position, color, direction, requiredThroughPoint)
```

對黑棋連珠規則，成五點必須形成正五，不能形成長連。

## 4. 四的定義

### 4.1 四（Four）

一組四顆同色棋，存在至少一個空點，落在該點可以形成五。

四可以是連續或有空隙，例如：

```text
. X X X X .
X X . X X
X . X X X
```

後兩者通常稱為跳四、嵌四或沖四形。

### 4.2 四的身分不能用「五格視窗」直接計數

一個活四：

```text
. X X X X .
```

左右兩端都能成五，但它仍然是**一個四**，不是兩個四。

因此程式必須用「四顆棋子的集合」識別一個四：

```text
FourKey = direction + sorted(stoneIndex[4])
```

同一組四顆棋即使出現在兩個五格視窗中，也只能計一次。

### 4.3 活四／直四（Straight Four、Free Four）

四顆棋連續，並且有兩個不同成五點：

```text
. X X X X .
```

程式語意：

```text
同一個 FourKey 對應 2 個不同成五點
```

活四通常無法由對手一手全部防住。

### 4.4 沖四／眠四（No-free Four、Closed Four）

只有一個成五點：

```text
O X X X X .
. X X . X X O
```

程式語意：

```text
同一個 FourKey 對應 1 個成五點
```

VCF 的中間攻擊通常就是這種只有唯一防點的四。

### 4.5 單線雙四（Line Double Four）

同一次落子在同一方向產生兩個不同的 `FourKey`。

不可只看方向數量。即使都在同一條線，只要是兩組不同四，也構成雙四。

### 4.6 四四（Double Four）

黑棋落下一子後，產生兩個以上不同的四，且這些四都包含本次落子。

可能是：

```text
兩個不同方向各有一個四
同一方向存在兩個不同四
同一方向一個四，另一方向另一個四
```

計數依據是不同的 `FourKey`，不是成五點數，也不是方向數。

## 5. 三的定義

### 5.1 表面三（Apparent Three）

從形狀看起來，有三顆同色棋，某個空點落下後可能形成活四。

例如：

```text
. . X X X . .
. X X . X . .
```

表面三只是候選，不能直接用來判三三。

### 5.2 真三／活三（True Three）

一組三顆同色棋，至少存在一個合法延伸點 `e`，使得：

1. 在 `e` 落子時不會立即形成五；
2. 落子後，在同一方向形成活四；
3. 若是黑棋，`e` 本身不是長連；
4. 若是黑棋，`e` 本身不是四四；
5. 若是黑棋，`e` 若表面上形成三三，該三三必須依相同規則遞迴驗證後屬於允許的三三。

只要有一個合法延伸點，原本這組三就是真三。

### 5.3 為什麼延伸成正五不算活三

三的定義要求「增加一子形成活四」，而不是直接形成五。

所以：

```text
某延伸點一落下就正五
```

不能用來證明原本是活三。

### 5.4 假活三

外觀像活三，但所有可形成活四的延伸點都非法或無效，例如：

```text
延伸點形成長連
延伸點形成四四
延伸點形成真正的三三禁手
延伸後不是活四
延伸時直接形成五
```

這種形不能算進三三。

### 5.5 三的身分

和四相同，不能單純用視窗數量計數。

建議：

```text
ThreeKey = direction + sorted(stoneIndex[3])
```

同一組三顆棋有多個延伸點，仍然只是一個三。

### 5.6 眠三、二、一

`眠三`、`活二`、`眠二`、`一` 常用於估值、排序與 VCT 搜尋，但不是連珠禁手的官方核心定義。

建議把它們標記為：

```text
heuristic pattern
```

不可用它們直接判斷三三。

## 6. 單線棋型分析

### 6.1 分析範圍

最安全做法是取出包含新落子點的整條線。

若使用局部視窗，至少要足以判斷：

```text
正五
長連
四
活四
三的活四延伸
```

原始 Evaluator 使用以落子點為中心的局部線段與滑動五格視窗。新版可使用整線、11 格局部編碼或 Bitboard，但結果必須一致。

棋盤外以 `#` 填充：

```text
# # . X @ X . # #
```

### 6.2 阻斷

以下內容會切斷同色棋列：

```text
對手棋
棋盤外
```

空點不會切斷「棋列候選」，但會切斷「連續棋列」。

### 6.3 建議的單線分析輸出

```cpp
struct LineAnalysis {
    bool exactFiveThroughMove;
    bool overlineThroughMove;

    vector<FourThreat> fours;
    vector<ThreeCandidate> apparentThrees;
};

struct FourThreat {
    Direction direction;
    array<int, 4> stones;          // FourKey
    vector<int> winningPoints;     // 1 個為沖四，2 個為活四
};

struct ThreeCandidate {
    Direction direction;
    array<int, 3> stones;          // ThreeKey
    vector<int> extensionPoints;   // 可能形成活四的點
};
```

### 6.4 四的通用產生法

對包含本次落子的同一條線：

1. 列舉線上的空點 `w`；
2. 暫時在 `w` 放入同色棋；
3. 檢查是否形成該規則允許的五；
4. 若是，找出形成該五的四顆原有棋；
5. 用 `FourKey` 合併；
6. 將 `w` 加入該四的成五點集合；
7. 還原棋盤。

最後：

```text
winningPoints.size == 1 → 沖四
winningPoints.size >= 2 → 活四
不同 FourKey 數量 >= 2 → 雙四
```

### 6.5 三的通用產生法

對包含本次落子的同一條線：

1. 找出三顆棋的候選集合；
2. 列舉空延伸點 `e`；
3. 暫時在 `e` 放同色棋；
4. 若 `e` 直接形成五，這個 `e` 不算活三延伸點；
5. 檢查同一方向是否形成活四；
6. 保存 `e` 到該 `ThreeKey`；
7. 還原棋盤。

這一步只產生表面三。黑棋延伸點是否合法，要由禁手遞迴再驗證。

## 7. 連珠黑棋禁手

### 7.1 適用範圍

```text
只有黑棋受禁手限制。
白棋沒有三三、四四、長連禁手。
```

非連珠規則時，`isForbiddenMove` 應直接回傳 `false`。

### 7.2 判斷只分析穿過新落子的四條線

黑棋下在 `p` 後，新產生的正五、長連、四、三都必須包含 `p`。

因此禁手判斷只需分析：

```text
穿過 p 的橫線
穿過 p 的直線
穿過 p 的兩條斜線
```

### 7.3 判斷優先順序

黑棋在空點 `p` 暫時落子後：

```text
1. 若任一方向形成正五：合法勝利，不判禁手
2. 否則若任一方向形成長連：長連禁手
3. 否則若不同 FourKey 總數 >= 2：四四禁手
4. 否則計算真正的 ThreeKey 數量
5. 真三數量 >= 2：三三禁手
6. 其他情況：合法
```

正五優先的理由是連珠規則的禁手條件以「黑棋沒有同時形成五」為前提。原始 Evaluator 也採用這個順序。

### 7.4 長連禁手

只要本次黑棋落子造成任一方向連續六顆以上，且沒有同時形成正五，便是禁手。

### 7.5 四四禁手

將四個方向產生的所有 `FourKey` 放入集合：

```text
uniqueFours
```

若：

```text
uniqueFours.size >= 2
```

則為四四禁手。

不可錯誤地寫成：

```text
有兩個成五點就是四四
```

因為活四本身有兩個成五點，但只是一個四。

### 7.6 三三禁手

先取得所有表面 `ThreeKey`，再逐一驗證。

對某個 `ThreeKey`：

```text
只要有一個 extensionPoint 能合法形成活四
→ 這個 ThreeKey 是真三
```

兩個以上不同真 `ThreeKey` 同時包含本次落子，才是三三禁手。

## 8. 真活三的遞迴驗證

### 8.1 規則原因

黑棋的活四延伸點 `e` 可能同時形成：

```text
長連
四四
另一個三三
```

前兩者可立即判非法；三三必須再確認其中有幾個是真三，所以會形成遞迴。

### 8.2 遞迴語意

```text
isTrueThree(T):
    對 T 的每個延伸點 e：
        暫時放黑棋 e

        若 e 直接形成五：
            e 不能證明 T 是活三

        否則若 e 形成長連：
            e 非法

        否則若 e 形成四四：
            e 非法

        否則：
            遞迴計算 e 產生的真三數量

            若真三數量 < 2：
                e 合法
                T 為真三

        還原 e

    所有 e 都失敗：
        T 為假三
```

### 8.3 遞迴終止

每深入一層都會在原本空點增加一顆黑棋，因此空點數嚴格減少，不可能無限建立相同局面。

仍建議加入：

```text
position hash + current point + analysis mode
```

作為快取鍵，避免不同路徑重複分析同一局面。

深度上限只能作為除錯保護，不可在超過上限時隨意判合法或判禁手，否則會破壞規則正確性。

### 8.4 避免重複落子

介面必須分清：

```cpp
isForbiddenMove(position, emptyPoint)
```

與：

```cpp
analyzeForbiddenAfterBlackPlaced(position, placedPoint)
```

建議：

```cpp
bool isForbiddenMove(Position& pos, int p) {
    putBlack(pos, p);
    bool result = analyzeForbiddenAfterBlackPlaced(pos, p);
    removeBlack(pos, p);
    return result;
}
```

遞迴內部只能呼叫「棋子已放置」版本，避免同一點重複放子。

### 8.5 建議偽程式

```text
isForbiddenMove(pos, p):
    assert p 為空
    放黑棋 p
    result = analyzePlacedBlack(pos, p)
    取回黑棋 p
    return result


analyzePlacedBlack(pos, p):
    lines = 分析 p 的四個方向

    if 任一方向為正五:
        return false

    if 任一方向為長連:
        return true

    fours = 合併四方向的不同 FourKey
    if fours 數量 >= 2:
        return true

    trueThreeCount = 0

    for each 不同 ThreeKey:
        if isTrueThree(pos, ThreeKey):
            trueThreeCount++

            if trueThreeCount >= 2:
                return true

    return false


isTrueThree(pos, three):
    for each e in three.extensionPoints:
        if e 非空:
            continue

        放黑棋 e
        line = 分析 e 在 three.direction 的棋型

        if e 沒有形成活四:
            取回 e
            continue

        if e 形成正五:
            取回 e
            continue

        illegal = analyzePlacedBlack(pos, e)
        取回 e

        if not illegal:
            return true

    return false
```

實作時要避免 `analyzePlacedBlack` 因正五優先而把「活三延伸時直接成五」誤當成合法延伸。必須先以「三的定義」排除直接成五，再檢查延伸點禁手。

## 9. VCF 定義

### 9.1 VCF 是什麼

VCF = Victory by Continuous Fours，中文常稱「連續沖四勝」。

攻方連續下出四的威脅，使守方每回合都必須處理立即成五點，直到攻方形成無法單手防住的終局威脅或直接勝利。

VCF 不等於任意強攻，也不等於 VCT。

```text
VCF：攻方每一步主要以四作為強制手段
VCT：可包含活三、四三等更廣泛的威脅
```

### 9.2 VCF 路線表示

建議路線格式：

```text
[攻0, 防0, 攻1, 防1, ..., 最終攻]
```

路線長度通常為奇數。

### 9.3 中間攻擊手

非終局攻擊手必須：

1. 落點合法；
2. 形成四；
3. 攻方只有一個立即成五點；
4. 守方因此有唯一必要防點。

若攻擊後有兩個以上無法同時防住的成五點，應視為終局，而不是繼續加入唯一防點。

### 9.4 守方強制防點

中間攻擊手形成沖四後：

```text
defense = 該四唯一的成五點
```

VCF 驗證器必須確認實際防守點等於此點，不能允許守方任意下其他位置。

### 9.5 VCF 終局

以下任一情況可作為成功終局，實際採用哪些應由規則模式明確指定：

```text
攻方已形成勝利五連
攻方形成活四，存在兩個不同成五點
攻方形成兩個以上不同四
白棋攻擊後，黑棋唯一防點是禁手（抓禁）
```

黑棋攻方的每一步仍必須合法；不能把黑棋禁手當成 VCF 勝著。

### 9.6 防守方的反沖四

守方在被迫防守時，落子可能同時形成自己的四。

因此正確的 VCF 搜尋不能只做：

```text
攻方沖四 → 守方擋住 → 攻方任意繼續
```

而應檢查：

```text
守方防守後是否產生立即成五威脅
```

若守方產生唯一的反沖四，攻方下一手必須同時處理該成五點，否則攻方會先輸。

原始 VCF 搜尋有處理這種反威脅；新版不可省略。

### 9.7 VCF 驗證器

搜尋器找到路線後，必須用獨立驗證器重播：

```text
清空搜尋暫存
從初始局面開始逐手重播
每手確認落點為空
每手確認顏色正確
黑棋每手確認非禁手
每個中間攻擊手確認為四
每個中間防守手確認是唯一強制防點
確認守方反沖四已被處理
確認最後一手構成合法終局
```

不得只相信搜尋器自己輸出的路線。

### 9.8 VCF 搜尋偽程式

```text
searchVCF(pos, attacker):

    if attacker 已合法獲勝:
        return success

    candidates = 產生所有合法四威脅手
    依威脅強度與後續候選數排序

    for attack in candidates:
        放 attack

        if attack 為終局:
            記錄路線
            取回 attack
            return success

        defense = attack 的唯一成五防點

        if defense 不存在或不可下:
            取回 attack
            continue

        放 defense

        counterThreat = 守方是否形成立即成五威脅

        if counterThreat 存在:
            下一攻擊手必須處理 counterThreat

        遞迴搜尋下一層

        取回 defense
        取回 attack

    return failure
```

實際搜尋還需要：

```text
最大深度
最大節點數
候選排序
```

## 13. 常見錯誤

### 錯誤 1：只列舉幾種字串棋型

禁手形狀很多，且棋盤邊界、跳空、重疊棋列會造成大量變形。應以語意產生四與三，再以集合去重。

### 錯誤 2：把活四的兩個成五點算成兩個四

活四只有一個 `FourKey`，即使有兩個成五點也不是四四。

### 錯誤 3：只數方向

同一方向可能形成兩個不同四，所以四四不能只用方向數判斷。

### 錯誤 4：把表面活三直接算成三

必須驗證延伸點是否合法，否則會把大量假三誤判成三三。

### 錯誤 5：延伸點直接成五也算活三

三必須延伸成活四，不能直接成五。

### 錯誤 6：忽略正五優先

黑棋同一步形成正五時，依連珠規則與原始 Evaluator 的實作，不應再以同一步的三三、四四或長連流程判禁手。

### 錯誤 7：棋盤外當空點

棋盤外是阻斷點，不是可下空點。

### 錯誤 8：遞迴時重複放子

必須區分「空點入口函式」與「棋子已放置分析函式」。

### 錯誤 9：提早返回前沒有還原棋盤

所有落子都必須一一回溯，否則後續節點會在錯誤局面上計算。

### 錯誤 10：VCF 忽略守方反沖四

守方被迫防守的棋也可能產生自己的立即威脅，攻方必須處理。

### 錯誤 11：搜尋結果不重播驗證

任何 VCF 路線都必須由獨立驗證器逐手確認。
