## 測試方法

- 使用一個沒有黑 VCF、但具有大量落子順序轉置的 `W.XXX.W` 組合盤面。
- 對盤面做 D4 的 8 種旋轉／鏡射，形成 8 個等價難盤。
- 每個版本、每個盤面重複 7 次，取中位耗時。
- 搜尋深度 41 ply，節點上限 1,500,000。
- 所有案例均完整搜尋結束：`routes=0`、`aborted_cases=0`、`valid=1`。
- 主要排名依據為 8 個盤面的中位耗時合計 `median_total_ms`；節點數與 nodes/s 只作輔助。
- Native 使用 `g++ -O3 -flto -DNDEBUG -std=c++17`。
- Wasm 使用 Emscripten 3.1.74、`-O3 -flto -msimd128`，由 Node.js 執行。

## 結論

本次 Native 與正式 LTO SIMD Wasm 的共同冠軍都是：

```text
65,536 buckets × 4 ways = 262,144 個表項
完整黑白 bitboard 精確比對
先對 Zobrist hash 做 SplitMix64 finalizer
四格循環的最近替換
ply 不參與命中 key
不保留淺層槽
```

關鍵觀察：

1. 四路 256K 幾乎保留了 `unordered_set` 的完整剪枝效果，但避開節點配置、bucket 與指標追蹤成本。
2. 小型直接表的單節點速度不差，但碰撞造成大量重搜；32K 比現行總耗時慢約 83%。
3. 原始 Zobrist 低位分布不穩定；128K 原始低位比同容量混合雜湊慢約 11.19%。
4. 「保留淺層＋最近」不是最佳替換策略；Wasm 四路淺層＋最近 256K 比單純四路最近替換慢約 3.62%。
5. 真正應採用的是總耗時冠軍，不是單看 nodes/s 最高的版本。

## 適用範圍

這是專門壓測完全同盤轉置的無解難盤。一般很快找到第一組 VCF 的局面，同型表作用較小，因此實際改善可能低於此 Native 數字；但 Native 與 Wasm 都由同一方案勝出，方向具有一致性。
