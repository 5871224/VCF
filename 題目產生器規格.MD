# VCF 題目產生器規格

> 本文件只描述目前正式運作的題目產生器。修改產生器時必須同步改寫本文件，並刪除被取代的舊規格。

## 1. 搜尋後端

題目產生器使用獨立 Bitboard Worker：

```text
makevcf-generator-core.js
  → rapfi/vcf-bitboard-worker.js
  → rapfi/engine/vcf-bitboard-engine.js / .wasm
```

規則：

- 不共用主分析頁的 Worker。
- 停止或重新啟動題目產生器時，不中止主分析。
- VCF 搜尋、路線精簡與防守點驗證使用 Bitboard Worker。
- 一般候選使用 `shortest`；「只保留目標 VCF」使用 `multi`。
- 兩種搜尋都依「多組剪枝」下拉選單使用 `fast` 或 `strict`。
- 頁面沒有剪枝下拉選單時，讀取 `vcf_multi_pruning`；沒有儲存值時預設 `fast`。
- 題目產生期間停用剪枝下拉選單。
- 模板枚舉使用所在頁面提供的同步棋型與禁手函式；`/rapfi/` 由 Bitboard 相容層提供。

## 2. VCF 步數

VCF 步數是**最終連五之前，攻方實際落子的手數**。

```text
攻方死四 → 守方擋 → 攻方活四 → 守方擋 → 攻方五連
```

以上為 2 步 VCF。最後連五那一手不計入。

同一手即使同時形成四四，也只算 1 步。程式必須重播手順計算，不得用形成的四數量代替落子手數。

題目可指定 1～10 步。

## 3. N 點

N 點有兩個獨立標記：

- `NO_BLACK`：黑方不能使用。
- `NO_WHITE`：白方不能使用。

同一點可同時具有兩種標記；判斷時依實際放入的棋色決定。

顯示：

- 綠色方塊：雙方 N 點。
- 紅框黑色方塊：黑方 N 點。
- 紅框白色方塊：白方 N 點。
- 方塊中心與棋盤交點重合。

完整 VCF 路線中的所有落點會加入雙方 N 點。有禁手且黑方進攻時，模板 X 點依黑方限制加入黑方 N 點。

## 4. 初始材料

初始材料分為活三與死四。

### 4.1 活三

```text
留 留 攻 攻 攻 留
留 攻 留 攻 攻 留
```

- 三個「攻」為攻方棋，皆可作 A。
- 三個「留」為雙方 N 點。
- 中間「留」為形成活四的攻擊點。
- 最外側兩個「留」為五連點。
- 活三接一層新死四後形成 2 步基礎。

### 4.2 死四

```text
X 攻 攻 攻 攻 留
攻 攻 攻 留 攻
攻 攻 留 攻 攻
```

死四一：

- X 為盤端或守方棋。
- X 在盤內時為攻方 N 點。
- 靠近 X 的攻方棋不能作 A。

死四二、三：

- 最外側兩顆攻方棋不能作 A。
- 靠近內部「留」的攻方棋可作 A。

所有「留」均為雙方 N 點。死四接一層新死四時，A 下回同時完成兩個四，形成 1 步 VCF。

有禁手且黑方進攻時，四四為禁手，因此不使用死四基礎；黑方有禁手模式無合法 1 步題目。

## 5. 新死四模板

```text
1. X 五 攻 攻 攻 攻 X
2. X 攻 五 攻 攻 攻 X
3. X 攻 攻 五 攻 攻 X
```

- `攻`：攻方棋。
- `五`：死四形成後守方唯一被迫防守點。
- `X`：模板外側位置。
- A 必須是盤面既有攻方棋，並與模板一個攻位重合。
- 完整套入模板後移除 A，A 成為新題目的第一手。
- 四個方向都可枚舉，不人工限制必須轉向。

## 6. 模板相容規則

### 6.1 攻方棋位置

可沿用既有攻方棋，或在不是攻方 N 點的空點新增攻方棋；不能覆蓋守方棋。

### 6.2 A 點

- 必須是原盤面既有攻方棋。
- 不能是初始材料標記為禁止 A 的棋。
- 不能是攻方 N 點。

### 6.3 五點

- 不能是攻方 N 點。
- 不能已有攻方棋。
- 可為空點或原本的守方棋。
- 原本是守方棋時，生成時暫時移除，VCF 過程中由守方下回。

### 6.4 X 點

X 可為空點、守方棋、N 點或盤端；不能已有攻方棋。

### 6.5 同線四四

新死四與原死四同方向時，A 下回必須形成單線四四，且該線恰有兩個不同五點。四四仍只算 1 步。

### 6.6 新死四產生活三

套入完整死四並移除 A 後，掃描新死四範圍。若攻方在某空點合法落子可於該方向形成活四，視為存在活三。

存在活三時，只能在左右 X 補守方棋；依序測試左、右、兩端。補後仍有活三則淘汰候選。補入的 X 必須是空點且不是守方 N 點。

## 7. 指定步數生成

### 7.1 1 步

1. 只使用死四基礎。
2. 在基礎外增加一層死四並移除 A。
3. A 下回必須合法形成四四。
4. 不得存在更短取勝路線。
5. 有禁手且黑方進攻時沒有合法候選。

### 7.2 2～10 步

1. 建立規則允許的活三與死四基礎。
2. 活三與死四皆可用時，先等機率選材料類別，再依該類候選權重抽選。
3. 活三加一層死四形成 2 步基礎。
4. 死四加一層死四形成 1 步基礎。
5. 每次延伸選盤面既有攻方棋作 A，再增加一層死四。
6. 每成功延伸一層，VCF 步數必須正好增加 1。
7. 候選失敗時回溯嘗試同層其他候選；整個基礎無法延伸時重建基礎。

層數關係：

```text
活三路線新增死四層數 = 目標步數 - 1
死四路線新增死四層數 = 目標步數
```

## 8. 標準完成盤面

延伸驗證不比較落子順序，只比較 225 格黑白棋位置。

標準完成盤面：

- 終局為四四：記錄四四形成後的盤面。
- 終局為活四：記錄攻方下出活四之前的盤面。
- 終局為連五：記錄連五形成後的盤面。
- 其他終局：記錄終局棋型完成時的盤面。

新增一層後，相較前一層可永久增加：

- 本層新增攻方棋。
- 封閉活三或封鎖較短 VCF 所加入的守方棋。
- 最後為補齊黑白子數加入的守方棋。

A 與原五點守方棋只是在生成時暫時移除，VCF 過程中下回，不算永久新增差異。

## 9. 剪枝選擇

每次題目搜尋送出前，重新讀取「多組剪枝」下拉選單：

```text
高速多組 VCF（集合子集） → pruning = fast
嚴格多組 VCF（完全同盤） → pruning = strict
```

- 一般候選、較短 VCF 補守、補齊黑白子數與唯一題檢查都使用目前選擇。
- 呼叫端殘留的舊 `pruning` 參數不得覆蓋下拉選單。
- 題目產生期間下拉選單停用，直到產生完成或停止。

## 10. 一般 VCF 驗證

一般候選使用新版：

```text
mode = shortest
pruning = 目前下拉選單
maxVCF = 64
maxDepth = 2 × 目標步數 + 1 ply
maxNode = 5,000,000
```

流程：

1. 搜尋目前最短的一批 VCF。
2. 重播每條路線，計算題目步數與標準完成盤面。
3. 若目前最短步數等於目標，至少一條路線必須符合預期完成盤面。
4. 若最短步數小於目標：
   - 未勾選「補齊黑白子數」時直接淘汰。
   - 勾選時進入較短 VCF 自動補守。
5. 沒有 VCF、最短步數高於目標或最短層沒有預期完成盤面時淘汰。

使用 `shortest` 的目的，是只處理目前真正需要先排除的最短層，不為每個候選列舉到 64 組才繼續。

## 11. 較短 VCF 自動補守

「補齊黑白子數」預設開啟，除了最後補齊子數，也負責封鎖比目標短的 VCF。

流程：

1. 對目前最短層每條較短 VCF 計算防守點。
2. 統計可同時封鎖較多較短路線的守方落點。
3. 排除 N 點、禁手、守方形成四、已占用點及受保護的目標防守點。
4. 加入一顆守方棋後，以相同下拉選單剪枝重新執行 `shortest`。
5. 重複直到最短層等於目標，並找到預期完成盤面。

限制：

- 每層最多嘗試 6 個防守點。
- 每個候選最多展開 42 個補守狀態。
- 目標路線已出現在目前最短層時，不可用會封鎖目標的點。
- 目標尚未出現在最短層時可先封鎖較短路線，但每個分支最後仍必須重新證明目標存在。
- 可加入的守方棋數不得超過目前黑白子數允許量。

## 12. 補齊黑白子數

目標題目完成後，依正常輪到攻方下棋所需的黑白子數補守方棋。

候選點規則：

- 必須在棋盤內圈。
- 必須為空點。
- 不能是守方 N 點。
- 不能是目前目標 VCF 的防守點。
- 守方落子不能形成四或連五。
- 有禁手且守方為黑棋時不得為禁手。

每補一顆守方棋後都以相同下拉選單剪枝重新執行 `shortest`，確認：

- 不出現比目標短的 VCF。
- 目標步數仍存在。
- 標準完成盤面等於原目標盤面加上該守方棋。

搜尋限制：

- 每層最多嘗試 10 個補子候選。
- 最多展開 140 個補子狀態。

補子權重：

- 攻方形成活四權重最高。
- 攻方形成死四次之。
- 攻守雙方形成活三或死三時乘上「三型加成」。
- 三型加成預設 30 倍。
- 再乘以該點周圍兩格內棋子數，使盤面較集中。

## 13. 只保留目標 VCF

勾選「只保留目標 VCF」時使用新版：

```text
mode = multi
pruning = 目前下拉選單
maxVCF = 64
maxDepth = 2 × 目標步數 + 1 ply
maxNode = 5,000,000
```

目標路線定義：

- 題目步數等於目標。
- 標準完成盤面等於預期盤面。

其他路線會以守方棋逐步封鎖：

- 每個狀態最多嘗試 8 個防守點。
- 最多展開 72 個封鎖狀態。
- 不可封鎖目標路線。

唯一性判定：

- 至少保留一條目標路線。
- 不能存在任何非目標路線。
- `aborted=true` 時不成立。
- 回傳 64 組時不成立。
- 達 5,000,000 節點時不成立。

達限制時沒有看到其他路線，不代表已證明唯一。

## 14. 候選權重

候選先完成枚舉、棋型篩選與去重，再依權重建立隨機順序。

```text
候選被排第一的機率 = 候選權重 ÷ 全部候選權重總和
```

### 14.1 沿用攻子

- 預設 10%。
- A 不計入沿用數。
- 計算新死四另外三個攻位中直接沿用盤面攻方棋的數量。
- 每沿用一顆增加一次 `99 × 百分比 ÷ 100`。

### 14.2 朝天元

- 預設 15%。
- 以新死四四個攻位計算平均位置。
- 朝向分數衡量 A 指向平均位置與 A 指向天元的夾角。
- 距離分數衡量平均位置接近天元的程度。
- 綜合分數為朝向分數乘距離分數。

### 14.3 最終權重

```text
候選權重
= 1
+ 沿用攻子數 × 99 × 沿用百分比 ÷ 100
+ 朝天元綜合分數 × 99 × 朝天元百分比 ÷ 100
```

權重只影響嘗試順序，不放寬 N 點、禁手、棋型、步數、完成盤面或其他 VCF 驗證。

## 15. 結果

產生成功後保存：

- 最終題目盤面。
- 攻方與規則。
- 目標 VCF 路線。
- 題目步數。
- 完成盤面與標準完成盤面。
- 每層 A、五點、方向、模板、永久新增攻守棋。
- N 點。
- 自動封鎖較短 VCF 的守方棋。
- 最後補齊子數的守方棋。
- 搜尋節點數與回傳組數。

「顯示答案」顯示目標路線；「顯示 N 點」切換 N 點圖層。

## 16. 驗收

每次修改至少確認：

1. 所有 `makevcf-generator-*.js` 通過語法檢查。
2. 題目產生器建立自己的 `rapfi/vcf-bitboard-worker.js`。
3. Worker 正確載入 `rapfi/engine/vcf-bitboard-engine.js`。
4. 主分析 Worker 不被題目產生器取消或重啟。
5. 一般驗證送出 `shortest`。
6. 「只保留目標 VCF」送出 `multi`。
7. 下拉選單選 `fast` 時送出高速剪枝，選 `strict` 時送出嚴格剪枝。
8. 題目產生期間剪枝下拉選單停用。
9. 深度、節點與 64 組限制正確傳入。
10. `getBlockVCF` 正確取得 `points`。
11. 比目標短的 VCF 能被淘汰或補守後重新驗證。
12. 目標步數與預期標準完成盤面均符合。
13. 達路線或節點限制時不誤判唯一。
14. 本文件、`AGENTS.md` 與 `新版Bitboard VCF規格.MD` 已同步更新。
