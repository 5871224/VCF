# VCF 分析工具規格書

文件版本：1.0  
文件日期：2026-07-16  
適用專案：`VCF`  
主要入口：`makevcf.html`

## 1. 文件目的

本文件定義 VCF 分析工具的產品目的、適用範圍、使用者功能、資料模型、搜尋介面、執行架構、圖片辨識流程、部署需求、限制與驗收標準，供後續開發、測試、部署及維護使用。

本文件以 `CLAUDE.md` 為原始說明基礎，並以目前程式碼實作為最終依據。若兩者內容不一致，應以 `makevcf.html`、`eval/`、`app/` 與現行執行檔的實際行為為準。

## 2. 專案目的

本專案提供連珠／五子棋 15×15 局面的 VCF（Victory of Continuous Four，連續衝四取勝）分析能力，讓使用者可在瀏覽器或桌面程式中完成下列工作：

1. 手動建立黑白棋局面。
2. 搜尋指定一方的 VCF 取勝路線。
3. 顯示多組 VCF、路線順序與搜尋效能。
4. 尋找單一路線及整體 VCF 的防守點。
5. 逐點試下，分析連五、四及可形成 VCF 的候選點。
6. 從圖片、截圖或手機相機匯入棋盤，辨識後套用至分析棋盤。
7. 在不傳送盤面或圖片到應用程式後端的前提下，於使用者裝置本機完成運算。

## 3. 產品定位與非目標

### 3.1 產品定位

- 定位為局面研究、題目製作與 VCF 路線分析工具。
- 主要服務對象為連珠／五子棋棋手、題目作者及分析人員。
- 核心輸出為 VCF 路線、候選點、防守點、節點數與搜尋速度。

### 3.2 非目標

- 不提供完整對局 AI 或全盤最佳著法評估。
- 不保證對所有局面完成數學上的勝負證明。
- 「VCT點」不是完整的 VCT 證明搜尋；目前是逐點試下後檢查連五、四或 VCF 的候選分析。
- 不提供帳號、雲端同步、多人協作或伺服器端棋局儲存。
- 圖片辨識為輔助輸入，不保證所有拍攝環境均可百分之百正確辨識。

## 4. 執行平台與架構

### 4.1 支援平台

| 平台 | UI | 搜尋後端 | 說明 |
|---|---|---|---|
| 一般瀏覽器 | `makevcf.html` | Web Worker → WebAssembly；不支援時改用 JavaScript Evaluator | 必須透過 HTTP/HTTPS 開啟，不支援直接使用 `file://` |
| Electron | `makevcf.html` | preload → IPC → `engine.exe` → Node → `engine_node.js` → JavaScript Evaluator | 主搜尋一個程序，逐點分析使用最多 8 個程序 |
| WebView2 | `makevcf.html` | WebView2 message bridge → `host.exe` → `engine.exe` | 僅在有建置 WebView2 host 時使用 |

### 4.2 瀏覽器資料流

```text
使用者操作
  → makevcf.html
  → VCFEngine 或 WorkerPool
  → eval/worker.js
  → EvaluatorWebassembly.js
  → Evaluator.wasm
  → 回傳 vcfInfo／候選點
  → SVG 顯示結果
```

### 4.3 桌面版資料流

```text
使用者操作
  → makevcf.html
  → window.engineAPI
  → preload.js
  → Electron IPC
  → app/main.js
  → engine.exe
  → engine_node.js
  → EvaluatorJScript.js
```

### 4.4 C++ 原生程式定位

`cpp/Searcher.cpp`、`cpp/Evaluator.cpp` 等原生搜尋程式仍保留於專案中，但目前 `app/engine.exe` 使用的是 web-parity Node 橋接路徑。規格與結果判讀不得把保留中的原生 C++ Searcher 當成現行 Electron 搜尋核心。

## 6. 棋盤與資料模型

### 6.1 棋盤

- 棋盤大小固定為 15×15，共 225 個交點。
- 一維索引公式：`idx = row * 15 + col`。
- `row`、`col` 的有效範圍為 0 至 14。

### 6.2 棋子值

| 值 | 意義 |
|---:|---|
| `0` | 空點 |
| `1` | 黑子 |
| `2` | 白子 |

### 6.3 引擎棋盤陣列

- UI 內部棋盤使用長度 225 的陣列。
- 傳給 Evaluator 時建立長度 226 的陣列，索引 225 設為 `-1` 作為邊界／相容標記。
- VCF 路線 `moves` 為交點索引陣列，第一手是攻方，之後攻守交替。

### 6.4 本機保存

- 使用 `localStorage` 自動保存棋盤。
- key：`vcf_board`。
- 格式：`{ "b": number[225], "nc": 1 | 2 }`。
- 重新開啟頁面時自動還原。
- 清空棋盤時刪除該 key。

## 7. 規則與棋型等級

### 7.1 規則

| UI 值 | 常數 | 意義 |
|---:|---|---|
| `1` | `GOMOKU_RULES` | 無禁手規則 |
| `2` | `RENJU_RULES` | 有禁手規則，預設值；黑棋檢查三三、四四及長連等禁手 |

切換規則時，主引擎及 Worker Pool 必須同步重建或重新設定規則。

### 7.2 現行 Web Evaluator 等級

| Level | 意義 |
|---:|---|
| `10` | 五連／勝點 |
| `9` | 活四 |
| `8` | 沖四 |
| `7` | 雙活三／雙 VCF 等複合威脅 |
| `6` | 活三／VCF 等級 |
| `4` | VCT 等級標記 |
| `0` | 無有效威脅 |

`getLevelPoint()` 的使用端通常以 `levelInfo & 0x0f` 取得低 4 位等級；高位可包含位置或額外標記。

## 8. 使用者功能需求

### FR-001 棋盤編輯

- 系統應以 SVG 顯示 15×15 棋盤及九個星位。
- 點擊空點時，依 `nextColor` 放置黑子或白子，之後切換下一色。
- 點擊已有棋子的交點時，應移除棋子，並將下一色改為被移除棋子的顏色。
- 每次異動後應保存棋盤至 `localStorage`。

### FR-002 規則選擇

- 使用者可選擇「有禁手」或「無禁手」。
- 預設為有禁手。
- 搜尋期間不得切換規則。

### FR-003 分析色選擇

- 使用者可選擇黑或白作為分析色。
- 分析色影響 VCF 防守、多組 VCF、VCT點、補黑及補白分析。
- 預設分析色為黑。

### FR-004 單組 VCF 搜尋

- 「黑子VCF」及「白子VCF」應搜尋指定顏色的一組 VCF。
- 空棋盤不得開始搜尋，狀態列應顯示「請先擺好棋型」。
- 預設參數：`maxVCF=1`、`maxDepth=200`、`maxNode=5,000,000`。
- 找到時應在棋盤依手順顯示序號，並記錄為最近一次 VCF 路線。
- 未找到時應清楚顯示未找到。
- 狀態列應顯示精確秒數、節點數與 `nodes/s`。

### FR-005 多組 VCF

- 應以 `maxVCF=20` 搜尋多組路線。
- 搜尋結果應執行 `trimVCFGroups`：
  1. 以完整手順重建局面。
  2. 若末手為活四，使用剪尾結果建立去重 key。
  3. 以落子集合及顏色正規化去重。
  4. 保留原始完整路線供 UI 顯示。
  5. 依路線長度由短至長排序。
- UI 應提供上一組／下一組切換。
- 狀態列應顯示唯一起點數、最短手數及處理前後組數。

### FR-006 單一路線防守點

- 必須先有最近一次 VCF 路線才能使用。
- 系統應呼叫 `getBlockVCF`，並包含對四的防守點。
- 結果以綠色三角形顯示。

### FR-007 全部 VCF 防守

- 系統應先確認分析色存在 VCF。
- 系統應取得該路線的防守候選點。
- 對每個候選點試下守方棋子，再以 Worker Pool 驗證攻方是否仍有 VCF。
- 使攻方不再有 VCF 的點才是有效防守點。
- 結果以藍色三角形顯示，並顯示節點數與 `nodes/s`。

### FR-008 VCT點候選分析

- 系統應先檢查目前局面是否已有分析色 VCF；若有，直接顯示該路線。
- 若目前沒有 VCF，系統應對所有空點試下分析色棋子。
- 每個試下點依序判斷：
  - 五連：label 為字串 `"5"`。
  - 活四或沖四：label 為字串 `"4"`。
  - 可形成 VCF：label 為數字型 VCF 路線手數。
- 本功能為候選點枚舉，不得對外宣稱為完整 VCT 證明。

### FR-009 補黑／補白找 VCF

- 「補黑找VCF」應在每個空點試下黑子，再搜尋分析色的 VCF。
- 「補白找VCF」應在每個空點試下白子，再搜尋分析色的 VCF。
- 瀏覽器及桌面版均應使用最多 8 個 Worker／程序分割空點並行運算。
- 顯示結果應包含五連、四、VCF 候選數、節點數及 `nodes/s`。

### FR-010 搜尋控制

- 搜尋期間應停用會產生競態的按鈕、規則及分析色選項。
- 「暫停」應中止主搜尋及 Worker Pool，並重新建立可用引擎。
- 「繼續」應以最近一次單組 VCF 搜尋的棋盤與顏色重新搜尋。
- 尚未搜尋時按繼續，應顯示提示。

### FR-011 清除功能

- 「清除路線/分析」只清除 VCF 路線及分析標記，不清除棋子。
- 「清空棋盤」清除棋子、路線、分析標記、最近搜尋參數及本機保存。

### FR-012 視覺化

- 已擺棋子顯示於 `stoneLayer`。
- VCF 路線顯示於 `vcfLayer`，包含攻守棋子與手順序號。
- 候選點及防守點顯示於 `analysisLayer`。
- VCF 與分析圖層不得攔截棋盤點擊事件。

### FR-013 分析標記樣式

| 結果 | 顯示 |
|---|---|
| 五連 label `"5"` | 粉紅色 `#ff66aa` |
| 四 label `"4"` | 橘褐色 `#c05000` |
| VCF 手數 | 由 `#7799ee` 至 `#001188` 依長短插值 |
| 最長 VCF 候選 | 額外顯示青藍色外環 `#00ccff` |
| 單一路線防守點 | 綠色三角形 `#00cc44` |
| 全部有效防守點 | 藍色三角形 `#4488ff` |

## 9. 圖片匯入與辨識

### 9.1 輸入方式

- 桌機檔案選擇。
- 剪貼簿貼上圖片。
- 行動裝置相機，使用 `capture="environment"` 優先開啟後鏡頭。

### 9.2 OpenCV.js

- 固定載入：`https://docs.opencv.org/4.13.0/opencv.js`。
- 僅在第一次需要圖片偵測或透視校正時動態載入。
- `ensureCvReady()` 最長等待 12 秒，直到 `cv.Mat` 可用。
- OpenCV 載入失敗時，應保留手動四角操作介面並顯示錯誤；透視校正不得繼續。

### 9.3 圖片正規化

- 圖片最大邊長限制為 1200 像素。
- 超過時應等比例縮小，避免過高記憶體及運算負擔。
- 原圖應縮放顯示於 520×520 的 source canvas 範圍內。

### 9.4 自動棋盤區域偵測

系統應依下列順序處理：

1. 將圖片轉為灰階。
2. 執行 5×5 Gaussian Blur。
3. 以門檻 50／150 執行 Canny。
4. 使用 `RETR_LIST` 及 `CHAIN_APPROX_SIMPLE` 尋找輪廓。
5. 將輪廓以周長 2% 的誤差近似多邊形。
6. 從四邊形中選取面積最大且至少占原圖 12% 的候選區域。
7. 將四點排序為左上、右上、右下、左下。
8. 找不到時改用預設內縮四角，讓使用者手動校正。

### 9.5 手動四角校正

- 使用者可用滑鼠或觸控拖曳四角。
- 四角不得超出目前圖片顯示範圍。
- 使用者確認後，系統應將四角映射回原圖座標。

### 9.6 透視校正

- 輸出大小固定為 900×900。
- 使用 `getPerspectiveTransform` 建立轉換矩陣。
- 使用 `warpPerspective`、線性插值及 `BORDER_REPLICATE` 產生校正棋盤。
- 在校正棋盤上均勻建立 15×15 共 225 個交點。

### 9.7 棋子分類

每個交點應取樣：

- 中心區域亮度及對比。
- 棋子環帶亮度、對比及梯度。
- 外圍背景亮度。
- 水平與垂直棋盤線可見比例。

系統應先判斷空點可能性與占用分數，再依全盤候選點環帶亮度的約 38% 與 68% 分位數建立黑白切分門檻。

輸出值：

| 值 | 分類 |
|---:|---|
| `0` | 空點 |
| `1` | 黑子 |
| `2` | 白子 |

- 信心值低於 0.65 時，預覽應加紅框。
- 套用後依黑白棋數推算下一手：黑子數小於等於白子數時輪黑，否則輪白。
- 使用者可在主棋盤手動修正辨識錯誤。

## 10. Worker 通訊規格

### 10.1 訊息格式

主頁傳送：

```javascript
{ cmd: "commandName", param: { /* command parameters */ } }
```

Worker 完成時回傳：

```javascript
{ cmd: "resolve", param: result }
```

### 10.2 命令

| cmd | 主要參數 | 回傳 |
|---|---|---|
| `setGameRules` | `{rules}` | 無資料 |
| `findVCF` | `{arr,color,maxVCF,maxDepth,maxNode}` | `vcfInfo` |
| `isVCF` | `{color,arr,moves}` | boolean |
| `getBlockVCF` | `{arr,color,vcfMoves,includeFour}` | `number[]` |
| `getBlockPoints` | `{arr,color,radius,maxVCF,maxDepth,maxNode}` | `number[]` |
| `selectPoints` | `{arr,color,radius,...}` | 棋盤標記陣列 |
| `selectPointsLevel` | `{arr,color,radius,...}` | 棋盤標記陣列 |
| `getLevelB` | `{arr,color,maxVCF,maxDepth,maxNode}` | `levelBInfo` |
| `getLevelPoints` | `{arr,color,placeColor,indices,maxDepth,maxNode}` | `{items,nodeCount}` |
| `trimVCFGroups` | `{arr,groups,color}` | 去重排序後的路線陣列 |

### 10.3 `vcfInfo`

至少包含：

| 欄位 | 型別 | 說明 |
|---|---|---|
| `vcfCount` | number | 找到的 VCF 數量 |
| `nodeCount` | number | 本次搜尋節點數 |
| `winMoves` | `number[][]` | VCF 路線集合 |
| `maxVCF` | number | 最大回傳路線數 |
| `maxDepth` | number | 最大搜尋深度 |
| `maxNode` | number | 最大節點數 |

### 10.4 `getLevelPoints` 回傳

```javascript
{
  items: [{ idx: number, label: "5" | "4" | number }],
  nodeCount: number
}
```

字串 `"5"`、`"4"` 必須與數字型 VCF 手數區分，避免五手 VCF 被誤判為立即五連。

## 11. 非功能需求

### NFR-001 純前端運算

- 瀏覽器版不得要求應用程式後端服務。
- 棋盤與匯入圖片不得由本專案主動上傳至遠端伺服器。
- OpenCV.js 為目前唯一按需下載的外部執行依賴。

### NFR-002 並行與效能

- Worker Pool 大小為 `min(hardwareConcurrency, 8)`，最低 1。
- Electron 程序池大小為 `min(os.cpus().length, 8)`，最低 1。
- 長時間搜尋應保持 UI 可操作，並可由暫停功能中止。
- 搜尋結果應以 `performance.now()` 計算時間。

### NFR-003 相容性

瀏覽器至少需要：

- ES2020 等級 JavaScript。
- Web Worker。
- WebAssembly；若沒有則使用 JavaScript Evaluator。
- SVG、Canvas 2D、File API 及 `localStorage`。
- 圖片辨識需要 OpenCV.js 4.13.0 可成功下載及初始化。

### NFR-004 可部署性

- 瀏覽器版應能部署至任意靜態 HTTP/HTTPS 主機，包括 GitHub Pages。
- 所有本地資源路徑必須保持相對路徑，以支援 `/<repository>/` 子路徑。
- 最小 Web 部署集合：
  - `makevcf.html`
  - `eval/`
  - `emoji/`
- 若要讓網站根網址直接開啟工具，部署來源頂層另需 `index.html`。

### NFR-005 桌面安全設定

- Electron 應保持 `contextIsolation: true`。
- Electron 應保持 `nodeIntegration: false`。
- Renderer 只能透過 preload 公開的 `engineAPI` 呼叫 IPC。

### NFR-006 錯誤處理

- Worker 建立失敗時應指出不可使用 `file://`。
- 搜尋例外應顯示於狀態列並寫入 console。
- 圖片偵測或透視失敗時應保留使用者可修正的狀態，不得直接覆蓋主棋盤。

## 12. 部署規格

### 12.1 一般靜態網站／GitHub Pages

```text
publish-root/
├─ index.html          # 建議；導向或等同 makevcf.html
├─ makevcf.html
├─ eval/
│  ├─ worker.js
│  ├─ Evaluator.js
│  ├─ EvaluatorWebassembly.js
│  ├─ EvaluatorJScript.js
│  └─ Evaluator.wasm
└─ emoji/
   └─ emoji.js
```

- 必須以 HTTPS 或 HTTP 提供。
- `Evaluator.wasm` 應可取得；目前程式以 `fetch` 讀取位元組後執行 `WebAssembly.instantiate`。
- GitHub Pages 專案站的相對路徑 `eval/...` 與 `emoji/...` 必須維持可達。

### 12.2 Electron

- 開發啟動：於 `app/` 執行 `npm start`。
- portable 建置：於 `app/` 執行 `npm run build`。
- electron-builder 輸出目錄：`C:/vcf_dist`。
- 套件應包含 `makevcf.html`、`eval/`、`emoji/`、`engine.exe` 及 `engine_node.js`。
- 現行 `engine.exe` 會呼叫系統 PATH 中的 `node`；目標電腦若未安裝 Node，桌面搜尋可能無法啟動。

## 14. 已知限制與風險

1. 搜尋器屬 VCF／威脅導向搜尋，不是完整五子棋證明引擎。
2. 「VCT點」是候選枚舉，不是完整 VCT 搜尋。
3. WebAssembly 與 JavaScript Evaluator 缺少自動化盤面回歸測試，版本更新時可能產生結果差異。
4. 圖片辨識依賴四邊形外框；外框被遮擋、嚴重裁切、非平面形變或強反光可能降低成功率。
5. 黑白空分類採亮度與棋盤線啟發式，應以真實照片資料集驗證準確率。
6. OpenCV.js 約 11 MB，第一次使用圖片匯入時可能受網路速度影響。
7. OpenCV 由外部官方網址載入；正式長期部署建議改為本地固定檔 `vendor/opencv.js`。
8. `localStorage` key 未包含專案命名空間；同一 origin 下其他工具若使用相同 key 可能互相覆蓋。
9. Electron portable 目前仍依賴系統 Node，並非完全自包含。
10. 專案目前沒有有效 Git 歷史，變更前應另行備份。
