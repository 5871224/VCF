# 新版 Bitboard VCF 規格

文件版本：1.2  
文件日期：2026-07-24  
適用入口：`/rapfi/`  
適用引擎：新版 C++ Bitboard WebAssembly

> [!IMPORTANT]
> 本文件只適用新版 `/rapfi/` Bitboard 工作台。
> `規格書.MD`、`CLAUDE.md` 既有內容及 `eval/` 下的說明主要描述舊版 Evaluator。
> 後續開發、除錯、效能測試或修改新版搜尋時，必須先讀本文件，不得直接把舊版架構、參數預設值或搜尋流程套到新版。

## 1. 文件目的與優先順序

本文件用來避免後續接手的 AI 或開發者混淆下列兩套系統：

1. 舊版 JavaScript／Evaluator WebAssembly。
2. 新版 C++ Bitboard WebAssembly。

處理新版 `/rapfi/` 功能時，文件與程式判讀優先順序如下：

1. 使用者在目前專案對話中明確確認的規則。
2. 本文件。
3. 新版實際程式碼與測試。
4. `題目產生器規格.MD` 中與棋理、步數、模板有關的共用規則。
5. `規格書.MD`、舊版 `CLAUDE.md` 與 `eval/` 舊引擎說明。

若本文件與舊版文件衝突，處理 `/rapfi/` 時以本文件為準。

## 2. 新舊版本分界

### 2.1 新版 Bitboard 主分析

新版 `/rapfi/` 主分析使用：

```text
/rapfi/ 頁面
  → rapfi/rapfi-bitboard-dashboard.js
  → rapfi/vcf-bitboard-main.js
  → rapfi/vcf-bitboard-worker.js
  → C++ Bitboard WebAssembly
```

頁面載入後另由相容層載入：

```text
rapfi/vcf-bitboard-generator-compat.js
  → rapfi/rapfi-workbench-header.js
  → rapfi/rapfi-question-bank.js
```

主要檔案：

| 路徑 | 用途 |
|---|---|
| `rapfi/vcf-bitboard-search-v2.cpp` | 新版搜尋組裝入口與正式匯出 |
| `rapfi/vcf-bitboard-search-fast-part1.inc` ～ `part4.inc` | 單組搜尋 DFS、候選產生與相容核心 |
| `rapfi/vcf-bitboard-search-single-tt-v5.inc` | 單組專用 256K 四路組相連精確同型表 |
| `rapfi/vcf-bitboard-search-exact-tt-v3.inc` | 多組嚴格／高速共用的完整盤面無解精確表 |
| `rapfi/vcf-bitboard-search-time-limit-v4.inc` | 多組時間／節點限制與停止原因寫入 |
| `rapfi/vcf-bitboard-search-multi-v3.inc` | 新版多組 VCF、嚴格／高速剪枝 |
| `rapfi/vcf-bitboard-worker.js` | Worker、Wasm ABI、路線輸出緩衝區與統計資料包裝 |
| `rapfi/vcf-bitboard-main.js` | 主執行緒引擎介面與 Worker 管理 |
| `rapfi/rapfi-bitboard-dashboard.js` | `/rapfi/` 搜尋選項、模式、結果與進度資訊 |
| `rapfi/rapfi-workbench-header.js` | 精簡頁首與強制重新整理按鈕 |
| `rapfi/rapfi-question-bank.js` | 瀏覽器本機題庫 |

### 2.2 舊版 Evaluator

下列檔案屬於舊版搜尋體系：

```text
eval/worker.js
eval/EvaluatorCore.js
eval/Evaluator.js
eval/EvaluatorWebassembly.js
eval/EvaluatorJScript.js
eval/Evaluator.wasm
```

舊版仍可用於：

- 舊頁面相容功能。
- 題目產生器目前的候選驗證。
- 新舊結果差異對照。

不得因舊版文件寫著 `maxVCF=20`、固定 500 萬節點或 `eval/worker.js`，就把新版 `/rapfi/` 改回相同架構。

### 2.3 題目產生器仍維持獨立舊版 Worker

題目產生器目前必須維持：

```text
makevcf-generator-core.js
  → eval/worker.js
```

即使頁面存在 `window.engineAPI`，也不得只因新版主分析完成就直接把題目產生器全面改走新版 Bitboard。

原因：題目產生器仍依賴舊版多組 VCF 的候選驗證語意。未完成專用差異測試、預期完成盤面驗證及較短 VCF 補守流程前，不得宣告遷移完成。

## 3. 不可破壞的核心原則

### 3.1 單組 VCF 優先保護

單組 VCF 使用專用 DFS 與單組四路精確同型表。

修改多組搜尋時：

- 不得讓單組搜尋改走多組列舉核心。
- 不得為了多組功能增加單組熱路徑的時間檢查或輸出處理。
- 必須保留找到第一組後立即返回。
- 多組嚴格／高速選項不得影響單組搜尋。
- 必須持續執行 Native 與 Wasm 單組 ABI／自我測試。

### 3.2 多組 VCF 強制精簡

新版 `multi` 與 `shortest` 固定執行路線精簡。

即使呼叫端傳入：

```js
simplify: false
```

多組與最短模式仍應精簡。`simplify` 參數只保留 ABI 相容用途。

### 3.3 多組處理順序

多組路線的正式順序為：

```text
找到原始路線
  → 先用原始手順做子集去重
  → 淘汰必定重複的延伸路線
  → 只精簡保留下來的路線
  → 精簡後排除完全相同輸出
  → 依手數排序
```

不得改成「每找到一組先精簡，再做子集去重」，否則會對最後仍被淘汰的路線反覆執行昂貴驗證。

### 3.4 搜尋剪枝與輸出子集去重是不同功能

必須明確區分：

- **搜尋剪枝**：決定 DFS 是否繼續展開目前盤面。
- **輸出子集去重**：路線已找到後，決定哪些代表路線保留給使用者。

嚴格模式不做「勝型子集搜尋剪枝」，但一般多組輸出仍必須執行既定的原始路線子集去重。

不得因嚴格模式只剪完全同盤，就取消多組結果的子集去重。

### 3.5 部分結果不得描述為完整列舉

當搜尋因 64 組、時間、節點或人工中止而停止：

- 已找到的路線必須保留。
- 不得把尚未完整搜尋的盤面寫入無解表。
- 不得把目前結果描述為全部 VCF。
- UI 必須提示結果可能不完整。

## 4. 搜尋模式

### 4.1 `single`

用途：最快找到一組 VCF。

特性：

- 找到第一組後立即返回。
- 使用 256K 四路組相連精確同型表。
- 不列舉所有路線。
- 預設最大深度 200 ply。
- 目前工作台呼叫的節點上限為 5,000,000。
- `pruning` 的嚴格／高速選項不得改變單組流程。

### 4.2 `multi`

用途：在指定最大深度內直接列舉多組 VCF。

特性：

- 不逐層加深。
- 直接使用呼叫端指定的 `maxDepth`。
- 強制精簡。
- 先子集去重，再精簡。
- 固定最多回傳 64 組。
- 達到 64 組、節點或時間限制時，保留已找到的部分結果。

### 4.3 `shortest`

用途：找最少手數的多組 VCF。

特性：

- 可逐奇數深度加深。
- 第一個找到結果的深度即為最短層。
- 強制精簡。
- 與一般 `multi` 不同，不得因題目產生器需要限制深度，就把一般 `multi` 改成逐層加深。

## 5. 單組 VCF 同型表

### 5.1 正式結構

單組使用 `rapfi/vcf-bitboard-search-single-tt-v5.inc`：

```text
65,536 buckets × 4 ways
= 262,144 個表項
```

規格：

- Zobrist hash 先經 SplitMix64 finalizer。
- 使用混合後 hash 的低 16 位定位 bucket。
- 每個 bucket 最多檢查 4 個表項。
- tag 相同後仍完整比較黑、白 bitboard。
- `ply`、攻擊層與落子順序不屬於命中 key。
- 四路全滿時，以 generation 內的循環 victim 選擇覆蓋其中一格。
- 覆蓋只會降低之後的命中率，不得造成錯誤剪枝。

### 5.2 共用表與 generation

舊單組外形仍會依 ply 建立多個容器，但這些容器必須共用同一張完整盤面表。

每次新搜尋：

- 只增加 generation。
- 不反覆配置整張表。
- 不每次清空全部表項。
- generation 回繞時才允許執行完整 generation 清理。

相同完整盤面即使由不同舊 ply 容器查詢，也必須命中。

### 5.3 精確性規則

單組同型表只可在下列條件全部成立時命中：

```text
目前黑棋完整 bitboard 完全相同；且
目前白棋完整 bitboard 完全相同。
```

不得只比較：

- bucket 索引。
- tag。
- 單一 64-bit hash。
- `ply`。

### 5.4 基準測試結論

2026-07-23 的 Native 與 LTO SIMD Wasm 完整搜尋基準中，256K 四路組相連、混合 hash、循環替換是共同最快方案。

後續若再修改表大小、路數或替換策略，必須以總耗時為主要指標，並同時記錄：

- 節點總數。
- nodes/s。
- 找到的路線。
- Native 與正式 LTO SIMD Wasm 結果。

不得只因 nodes/s 較高就判定總搜尋一定更快。

## 6. 多組剪枝

### 6.1 嚴格剪枝 `strict`

嚴格模式的搜尋剪枝條件只有一個：

```text
目前盤面的完整黑棋位置，與已記錄盤面的黑棋位置完全相同；且
目前盤面的完整白棋位置，與已記錄盤面的白棋位置完全相同。
```

`ply`、攻擊層數與落子順序不屬於同型表 key。

只有已完整搜尋並證明無解的盤面才能寫入嚴格同型表。搜尋因組數、節點、時間或其他限制中止時，不得把未完成盤面記為無解。

嚴格模式不用檢查已知勝型是否為目前盤面的子集。

### 6.2 多組精確同型表

多組搜尋使用 `rapfi/vcf-bitboard-search-exact-tt-v3.inc` 的固定容量直接對映表。

目前規格：

- 131,072 個表項。
- 每個表項保存完整黑 bitboard、完整白 bitboard 與非零雜湊 tag。
- 固定記憶體約 9 MiB。
- 不依 `ply` 建立多個 bucket。
- 不使用 `unordered_set`、節點配置或執行中的 rehash。
- 每次查詢只定位一個表項。

雜湊只負責定位與快速篩選。真正命中前仍必須逐字比較完整黑、白 bitboard。

表項碰撞時可直接覆蓋舊項目：

- 覆蓋只會讓之後少命中一次剪枝。
- 完整盤面不同時絕不可命中。
- 碰撞不得造成錯誤剪枝。

### 6.3 高速剪枝 `fast`

高速模式包含完整同盤無解剪枝，並額外使用已完整證明有解的黑白棋集合子集剪枝。

只有在某盤面已完整搜尋完成後，才能把它記錄為勝型子集。

若已證明勝型的：

- 黑棋集合是目前盤面黑棋集合的子集；且
- 白棋集合是目前盤面白棋集合的子集；

則目前分支最後只會形成被既有較短路線子集去重淘汰的延伸路線，可直接略過。

未完整搜尋的盤面不得加入高速剪枝資料。

## 7. 多組搜尋限制

### 7.1 組數固定上限

目前多組輸出緩衝區固定最多容納 64 組：

```text
MAX_ROUTES = 64
```

介面不提供組數設定欄位。

不得把「不顯示組數設定」誤解為「組數完全不限制」。真正取消 64 組上限前，必須先重設 Worker／Wasm 路線緩衝區、ABI 與記憶體配置。

### 7.2 時間與節點設定

工作台只提供兩個多組限制欄位：

| 欄位 | 單位 | 預設 | `0` 的意義 |
|---|---:|---:|---|
| 時間限制 | 秒 | 30 | 不限制時間 |
| 節點限制 | 百萬節點 | 20 | 不限制節點 |

設定會保存在 `localStorage`：

```text
vcf_multi_time_seconds
vcf_multi_node_millions
```

介面允許範圍：

- 時間：0～2,097,151 秒。
- 節點：0～1,023 百萬節點。

### 7.3 既有 `maxNode` 參數的封裝格式

為避免破壞 V3 ABI，多組把時間與節點封裝在既有 32-bit `maxNode` 參數：

```text
bit 31      ：封裝標記
bit 0..9    ：節點上限，單位百萬；0 = 不限制
bit 10..30  ：時間上限，單位秒；0 = 不限制
```

單組仍把 `maxNode` 當一般節點整數，不使用此封裝語意。

### 7.4 時間檢查頻率

不得每個節點都呼叫系統時間函式。

正式規格為：

```text
每 524,288 個節點檢查一次時間
```

等價條件：

```cpp
(nodes & ((1u << 19) - 1)) == 0
```

因此時間停止可能略晚於設定秒數；延遲取決於目前 nodes/s。

節點限制仍使用便宜的整數比較，且每個節點都可判斷。

### 7.5 停止與部分結果

任一限制先到即停止：

- 64 組回傳上限。
- 時間限制。
- 節點限制。
- 人工中止或其他中止條件。

停止後：

- 保留已找到路線。
- `aborted` 必須反映搜尋未完整。
- C++ 統計的保留欄位會記錄時間或節點停止原因。
- 目前 Worker `readStats()` 尚未把保留欄位公開成 `stopReason`；UI 依 `aborted`、節點數、時間設定及組數判斷提示。

未來若公開 `stopReason`，建議語意：

```text
complete
routes
time
nodes
cancelled
```

不得只靠單一 `aborted=true` 長期維持模糊判斷。

## 8. 結果與進度資訊

搜尋進行中至少顯示：

- 分析色。
- 高速或嚴格剪枝。
- 已執行秒數。
- 固定最多 64 組。
- 時間與節點限制。

搜尋完成後至少顯示：

- 唯一起點數。
- 最短手數。
- 最終組數。
- 節點數。
- nodes/s。
- 是否達到 64 組、時間或節點限制。
- 搜尋是否完整。

注意：主頁面的 `setInterval` 只負責更新 UI，不等於 C++ 內部時間限制。正式停止條件必須由 C++ 搜尋核心執行。

## 9. `/rapfi/` 介面規格

### 9.1 搜尋選項

目前搜尋選項包含：

- 精簡手順：主要影響單組；多組與最短固定精簡。
- 多組剪枝：`fast` 或 `strict`。
- 時間限制：秒。
- 節點限制：百萬節點。
- 補子搜尋：單組速度或多組最少步。

搜尋執行中，上述設定必須停用，避免執行途中改變語意。

### 9.2 精簡頁首

原本的工作台標題、長說明、四個架構狀態卡與引擎狀態列已移除。

頁首只保留：

1. `Rapfi 官方對照／棋型實驗室`。
2. `強制重新整理`。

不得在未經確認下重新加回大型說明卡片。

### 9.3 強制重新整理

網頁無法直接送出瀏覽器原生 `Ctrl＋F5`，目前按鈕採用可控制範圍內最接近的流程：

1. 清除同網域 Cache Storage。
2. 蒐集目前已載入的同網域資源。
3. 以 `fetch(..., { cache: "reload" })` 重新取得資源。
4. 在目前網址加入 `_refresh=<時間戳>`。
5. 使用 `location.replace()` 重新載入。

強制重新整理不得清除 `localStorage`，因此下列資料會保留：

- 目前盤面。
- 多組剪枝與限制設定。
- 題庫。

### 9.4 本機題庫

題庫由 `rapfi/rapfi-question-bank.js` 實作，資料保存在目前瀏覽器的 `localStorage`。

儲存鍵：

```text
vcf_question_bank_v1
vcf_question_bank_index_v1
```

每題保存：

```js
{
  board: [225 格黑白空盤面],
  nextColor: 1 | 2,
  addedAt: 時間戳
}
```

題庫資料不會自動同步到其他手機、電腦、瀏覽器或無痕模式。

### 9.5 題庫操作

題庫區塊包含：

- `加入題庫`。
- `上一題`。
- `下一題`。
- `刪除`。
- 題號顯示。

題號文字：

```text
題庫 0 題
第 N / 總數 題
未選題／共 N 題
```

操作規則：

- 空白盤面不得加入。
- 完整 225 格盤面相同時視為同一題，不重複新增。
- 重複加入時直接定位到既有題號。
- 切換題目時載入完整盤面與下一手顏色。
- 切換前清除舊 VCF 路線、分析標記、多組索引及題目答案圖層。
- 手動修改、匯入或清空盤面後，題號必須重新比對。
- 目前盤面不屬於題庫時顯示「未選題」。
- 刪除前顯示確認視窗。
- 刪除後優先載入原位置的新題目；刪除最後一題時載入上一題。
- 題庫清空後保留目前棋盤。
- 搜尋執行中停用題庫按鈕，避免 Worker 回傳舊盤面結果。

手機版題庫按鈕使用兩欄排列，題號獨立占一整列。

## 10. 題目產生器未來接新版時的專用規則

本節是已確認的遷移方向，目前尚未完成全面切換。

### 10.1 一般層驗證

- 不逐層加深。
- 搜尋深度直接限制為本層目標深度。
- 守方應對點唯一時，每兩手可記錄一次同型表。
- 子集去重維持不變。
- 若預期完成盤面被去重移除，改換一個死四候選。

### 10.2 達到限制時

當滿 64 組、達節點限制或達時間限制時：

1. 檢查預期完成盤面是否只是被子集去重去掉。
2. 若存在較短 VCF，可先補一顆守子防守該較短 VCF。
3. 補守後重新驗證候選。
4. 不得單純因回傳部分結果就把候選視為完整通過。

### 10.3 「只保留目標 VCF」

若勾選「只保留目標 VCF」：

- 前面各層仍只搜尋到目標深度，避免重複搜尋。
- 到最後一個死四時，再把搜尋深度開到最大。
- 若達到組數、節點或時間限制，且存在其他 VCF，可補守子封鎖其他 VCF 後重新驗證。

### 10.4 切換條件

題目產生器只有在以下項目完成後才能宣告全面改用新版：

- 新版可正確辨識預期完成盤面。
- 新版可處理被子集去重的預期盤面。
- 較短 VCF 補守流程完成。
- 限制中止原因可被題目流程可靠判讀。
- 舊版與新版對相同候選的合格／不合格判定完成差異測試。

## 11. API 與回傳資料

新版 `findVCF` 主要參數：

```js
{
  arr,
  color,
  rules,
  mode: "single" | "multi" | "shortest",
  pruning: "strict" | "fast",
  simplify,
  maxVCF,
  maxDepth,
  maxNode
}
```

目前多組的時間與節點限制封裝在 `maxNode`，尚未新增獨立 `maxTimeMs` ABI 參數。

主要回傳欄位：

```js
{
  winMoves,
  vcfCount,
  nodeCount,
  elapsedMs,
  nodesPerSecond,
  routeCount,
  candidateCount,
  maxPly,
  aborted,
  searchMode,
  pruning,
  simplified
}
```

目前 Worker 尚未公開 C++ 統計中的停止原因保留位。若未來新增：

```js
stopReason: "complete" | "routes" | "time" | "nodes" | "cancelled"
```

必須同步更新：

- Worker `readStats()`。
- 主執行緒結果型別。
- UI 提示。
- Native／Wasm ABI 測試。

## 12. 修改與驗收規則

每次修改新版搜尋至少驗證：

1. Native C++ 編譯成功。
2. LTO SIMD Wasm 編譯成功。
3. 棋型與禁手自我測試成功。
4. 單組正式 ABI 測試成功。
5. 單組四路表跨舊 ply 容器仍可命中相同盤面。
6. 單組新 generation 不得讀到上一輪資料。
7. 相同 hash、但完整黑白盤面不同時不得命中。
8. 嚴格與高速剪枝確實走不同邏輯。
9. 多組嚴格同型表只在完整黑白盤面相同時命中。
10. 高速模式不使用未完整搜尋的勝型子集。
11. 多組仍為先原始子集去重、再精簡。
12. 64 組、時間或節點停止時仍保留已找到結果。
13. 時間只每 524,288 節點檢查一次。
14. UI 的時間／節點 `0` 正確表示不限制。
15. 題目產生器仍使用獨立舊版 Worker，除非該次工作明確是執行遷移。

介面修改至少驗證：

- JavaScript 語法檢查成功。
- 手機版不超出畫面。
- 強制重新整理不清除 `localStorage`。
- 題庫加入、去重、上一題、下一題、刪除與題號同步正確。
- 切換題目會清除舊路線及分析圖層。
- 搜尋期間題庫操作停用。

效能修改需以相同盤面比較：

- 總耗時。
- 節點數。
- nodes/s。
- 找到的 VCF 手順。
- Native 與正式 LTO SIMD Wasm。
- 冷啟動與重複執行中位數。

## 13. 給後續 AI 的禁止事項

處理新版 `/rapfi/` 時，不得：

- 只讀舊版 `CLAUDE.md` 就判斷現行搜尋架構。
- 把 `eval/worker.js` 當成新版主分析 Worker。
- 把舊版 `maxVCF=20` 當成新版多組上限。
- 把「沒有組數設定欄位」解讀為無限回傳組數。
- 為了修多組而替換單組專用 DFS 或四路同型表。
- 關閉多組強制精簡。
- 把處理順序改成先精簡再子集去重。
- 把嚴格搜尋剪枝與多組輸出子集去重混為一談。
- 在精確同型表中只比較 hash，而不驗證完整黑白 bitboard。
- 把 `ply` 當成同型命中 key。
- 每個節點都讀取系統時間。
- 把 524,288 節點時間檢查改回高頻率而未先做效能測試。
- 把達到限制後的部分結果描述為完整列舉。
- 在未完成題目專用驗證前，把題目產生器全面重新導向新版。
- 讓強制重新整理清除題庫或其他 `localStorage` 設定。
- 把本機題庫描述成跨裝置雲端同步題庫。
- 未經確認重新加回大型工作台說明區塊。
