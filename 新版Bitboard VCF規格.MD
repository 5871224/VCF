# 新版 Bitboard VCF 規格

文件版本：1.1  
文件日期：2026-07-23  
適用入口：`/rapfi/`  
適用引擎：新版 C++ Bitboard WebAssembly

> [!IMPORTANT]
> 本文件只適用新版 `/rapfi/` Bitboard 工作台。
> `規格書.MD`、`CLAUDE.md` 既有內容及 `eval/` 下的說明主要描述舊版 Evaluator。
> 後續開發、除錯、效能測試或修改新版搜尋時，必須先讀本文件，不得直接把舊版架構、參數預設值或搜尋流程套到新版。

---

## 1. 文件目的與優先順序

本文件用來避免後續接手的 AI 或開發者混淆下列兩套系統：

1. 舊版 JavaScript／Evaluator WebAssembly。
2. 新版 C++ Bitboard WebAssembly。

處理新版 `/rapfi/` 功能時，文件與程式判讀優先順序如下：

1. 使用者在目前專案對話中明確確認的規則。
2. 本文件。
3. 新版實際程式碼與測試。
4. `題目產生器規格.MD` 中與棋理、步數、模板有關的共用規則。
5. `規格書.MD`、舊版 `CLAUDE.md` 與 `eval/` 舊引擎說明。

若本文件與舊版文件衝突，處理 `/rapfi/` 時以本文件為準。

## 2. 新舊版本分界

### 2.1 新版 Bitboard 主分析

新版 `/rapfi/` 主分析使用：

```text
/rapfi/ 頁面
  → rapfi/rapfi-bitboard-dashboard.js
  → rapfi/vcf-bitboard-main.js
  → rapfi/vcf-bitboard-worker.js
  → C++ Bitboard WebAssembly
```

主要搜尋檔案：

| 路徑 | 用途 |
|---|---|
| `rapfi/vcf-bitboard-search-v2.cpp` | 新版搜尋組裝入口 |
| `rapfi/vcf-bitboard-search-fast-part1.inc` ～ `part4.inc` | 目前最快的單組相容核心 |
| `rapfi/vcf-bitboard-search-exact-tt-v3.inc` | 多組專用完整盤面精確同型表 |
| `rapfi/vcf-bitboard-search-multi-v3.inc` | 新版多組 VCF、嚴格／高速剪枝 |
| `rapfi/vcf-bitboard-worker.js` | Worker 與 C++ Wasm ABI 包裝 |
| `rapfi/vcf-bitboard-main.js` | 主執行緒引擎介面與 Worker 管理 |
| `rapfi/rapfi-bitboard-dashboard.js` | `/rapfi/` 介面、模式選擇、結果與進度資訊 |

### 2.2 舊版 Evaluator

下列檔案屬於舊版搜尋體系：

```text
eval/worker.js
eval/EvaluatorCore.js
eval/Evaluator.js
eval/EvaluatorWebassembly.js
eval/EvaluatorJScript.js
eval/Evaluator.wasm
```

舊版仍可用於：

- 舊頁面相容功能。
- 題目產生器目前的候選驗證。
- 新舊結果差異對照。

不得因舊版文件寫著 `maxVCF=20`、固定 500 萬節點或 `eval/worker.js`，就把新版 `/rapfi/` 改回相同架構。

### 2.3 題目產生器目前仍是舊版 Worker

題目產生器目前必須維持獨立的：

```text
makevcf-generator-core.js
  → eval/worker.js
```

即使頁面存在 `window.engineAPI`，題目產生器也不得自動改走新版 Bitboard。

原因：題目產生器仍依賴舊版多組 VCF 的候選驗證語意。新版主分析修正完成，不代表題目產生器已完成遷移。

## 3. 不可破壞的核心原則

### 3.1 單組 VCF 優先保護

單組 VCF 使用目前最快的舊版相容 C++ Bitboard 核心。

修改多組搜尋時：

- 不得讓單組搜尋改走較慢的多組列舉核心。
- 不得為了多組功能重寫單組熱路徑。
- 必須保留單組找到第一組後立即返回的行為。
- 必須持續執行 Native 與 Wasm 單組 ABI／自我測試。

### 3.2 多組 VCF 強制精簡

新版多組模式固定執行路線精簡。

即使呼叫端傳入：

```js
simplify: false
```

多組與最短模式仍應精簡。`simplify` 參數只保留 ABI 相容用途。

### 3.3 多組處理順序

多組路線的正式順序為：

```text
找到原始路線
  → 先用原始手順做子集去重
  → 淘汰必定重複的延伸路線
  → 只精簡保留下來的路線
  → 精簡後排除完全相同輸出
  → 依手數排序
```

不得改成「每找到一組先精簡，再做子集去重」，否則會對最後仍被淘汰的路線反覆執行昂貴驗證。

### 3.4 搜尋剪枝與輸出子集去重是不同功能

必須明確區分：

- **搜尋剪枝**：決定 DFS 是否繼續展開目前盤面。
- **輸出子集去重**：路線已找到後，決定哪些代表路線保留給使用者。

嚴格模式不做「勝型子集搜尋剪枝」，但一般多組輸出仍必須執行既定的原始路線子集去重。

不得因嚴格模式只剪完全同盤，就取消多組結果的子集去重。

### 3.5 子集去重規則有效

較短路線的攻守棋集合若包含於較長路線中，較長路線可被淘汰。

這是既定規格，不是錯誤。若題目產生器未來使用新版，而預期完成盤面因子集去重消失，應改換該層死四或依題目規則補守子，不得取消一般多組的子集去重。

## 4. 搜尋模式

### 4.1 `single`

用途：最快找到一組 VCF。

特性：

- 找到第一組後立即返回。
- 使用原本最快的單組固定式勝負同型表。
- 不列舉所有路線。
- 多組剪枝選項不得影響單組結果或速度。

### 4.2 `multi`

用途：在指定最大深度內直接列舉多組 VCF。

特性：

- 不逐層加深。
- 直接使用呼叫端指定的 `maxDepth`。
- 強制精簡。
- 先子集去重，再精簡。
- 達到組數、節點或時間限制時，保留已找到的部分結果並標示搜尋未完整。

### 4.3 `shortest`

用途：找最少手數的多組 VCF。

特性：

- 可逐奇數深度加深。
- 第一個找到結果的深度即為最短層。
- 與一般 `multi` 不同，不得因題目產生器需要限制深度，就把一般 `multi` 改成逐層加深。

## 5. 多組剪枝

### 5.1 嚴格剪枝 `strict`

嚴格模式的搜尋剪枝條件只有一個：

```text
目前盤面的完整黑棋位置，與已記錄盤面的黑棋位置完全相同；且
目前盤面的完整白棋位置，與已記錄盤面的白棋位置完全相同。
```

`ply`、攻擊層數與落子順序不屬於同型表 key。

原因：

- 起始盤固定。
- 搜尋過程沒有提子或移除棋子。
- 每深入一層固定增加攻方一子與守方一子。
- 因此黑白棋位置完全相同時，新增棋子數與攻擊層數自然相同。

只有已完整搜尋並證明無解的盤面才能寫入嚴格同型表。搜尋因組數、節點、時間或其他限制中止時，不得把未完成盤面記為無解。

嚴格模式不用檢查已知勝型是否為目前盤面的子集。

嚴格模式用於：

- 結果正確性對照。
- 驗證高速剪枝是否漏掉不應漏掉的代表性路線。

### 5.2 嚴格模式專用精確同型表

多組搜尋使用 `rapfi/vcf-bitboard-search-exact-tt-v3.inc` 的固定容量直接對映表。

目前規格：

- 131,072 個表項。
- 每個表項保存完整黑 bitboard、完整白 bitboard與非零雜湊 tag。
- 固定記憶體約 9 MiB。
- 不依 `ply` 建立多個 bucket。
- 不使用 `unordered_set`、節點配置或執行中的 rehash。
- 每次查詢只定位一個表項。

雜湊只負責定位與快速篩選。真正命中前仍必須逐字比較完整黑、白 bitboard。

表項碰撞時可直接覆蓋舊項目：

- 覆蓋只會讓之後少命中一次剪枝。
- 不得只因 tag 或單一 64-bit 雜湊相同就剪枝。
- 完整盤面不同時絕不可命中，因此碰撞不會造成錯誤剪枝。

此專用表只替換新版多組核心使用的同型表，不得改動單組熱路徑。

### 5.3 高速剪枝 `fast`

高速模式包含完整同盤無解剪枝，並額外使用已完整證明有解的黑白棋集合子集剪枝。

只有在某盤面已完整搜尋完成後，才能把它記錄為勝型子集。

若已證明勝型的：

- 黑棋集合是目前盤面黑棋集合的子集；且
- 白棋集合是目前盤面白棋集合的子集；

則目前分支最後只會形成被既有較短路線子集去重淘汰的延伸路線，可直接略過。

未完整搜尋的盤面不得加入高速剪枝資料。

## 6. 搜尋限制

### 6.1 目前已實作

目前一般多組 VCF：

- 最多 64 組。
- 最大深度 200 ply。
- 最大節點目前為 5,000,000。
- 搜尋中顯示經過時間。
- 達到 64 組或節點上限時，結果列明搜尋未完整。

上述值是目前實作狀態，不代表長期固定規格。

### 6.2 已確認的下一步需求：三種限制可獨立設定

新版多組 VCF 應提供：

1. 限制組數。
2. 限制時間。
3. 限制節點。

任一限制先到即停止，並保留已找到路線。

建議介面語意：

| 設定 | 建議行為 |
|---|---|
| 組數 | 1～64；目前 Wasm 輸出緩衝區上限為 64 |
| 時間 | 秒；`0` 代表不限制時間 |
| 節點 | 正整數；`0` 可代表不限制節點，但 C++ 內部仍需使用安全上限 |

停止後必須回傳或顯示明確原因：

- 已達組數限制。
- 已達時間限制。
- 已達節點限制。

### 6.3 時間限制的效能規則

不得每個節點都呼叫高精度時間函式。

建議每 4096 個節點檢查一次時間：

```cpp
nodes++;
if ((nodes & 4095) == 0) {
    checkTimeLimit();
}
```

組數限制只在加入路線或進入主要分支時檢查；節點限制沿用便宜的整數比較。新增限制不得明顯降低無限制搜尋速度。

## 7. 結果與進度資訊

搜尋進行中至少顯示：

- 分析色。
- 高速或嚴格剪枝。
- 已執行秒數。
- 組數、時間、節點上限。

搜尋完成後至少顯示：

- 唯一起點數。
- 最短手數。
- 最終組數。
- 節點數。
- nodes/s。
- 是否達到任一限制。
- 搜尋是否完整。

注意：主頁面的計時器只負責更新 UI，不等於 C++ 內部時間限制。正式時間停止條件必須由 Worker 中的 C++ 搜尋核心執行。

## 8. 題目產生器未來接新版時的專用規則

本節是已確認的遷移方向，目前尚未切換。

### 8.1 一般層驗證

- 不逐層加深。
- 搜尋深度直接限制為本層目標深度。
- 守方應對點唯一時，每兩手可記錄一次同型表。
- 子集去重維持不變。
- 若預期完成盤面被去重移除，改換一個死四候選。

### 8.2 達到限制時

當滿 64 組、達節點限制或未來達時間限制時：

1. 檢查預期完成盤面是否只是被子集去重去掉。
2. 若存在較短 VCF，可先補一顆守子防守該較短 VCF。
3. 補守後重新驗證候選。
4. 不得單純因回傳部分結果就把候選視為完整通過。

### 8.3 「只保留目標 VCF」

若勾選「只保留目標 VCF」：

- 前面各層仍只搜尋到目標深度，避免重複搜尋。
- 到最後一個死四時，再把搜尋深度開到最大。
- 若達到組數、節點或時間限制，且存在其他 VCF，可補守子封鎖其他 VCF後重新驗證。

### 8.4 切換條件

題目產生器只有在以下項目完成後才能由舊 Worker 切換到新版：

- 新版可正確辨識預期完成盤面。
- 新版可處理被子集去重的預期盤面。
- 較短 VCF 補守流程完成。
- 限制中止原因可被題目流程可靠判讀。
- 舊版與新版對相同候選的合格／不合格判定完成差異測試。

## 9. API 與回傳資料

新版 `findVCF` 主要參數：

```js
{
  arr,
  color,
  rules,
  mode: "single" | "multi" | "shortest",
  pruning: "strict" | "fast",
  simplify,
  maxVCF,
  maxDepth,
  maxNode
}
```

未來加入時間限制時，建議新增：

```js
maxTimeMs
```

主要回傳欄位：

```js
{
  winMoves,
  vcfCount,
  nodeCount,
  elapsedMs,
  nodesPerSecond,
  routeCount,
  candidateCount,
  maxPly,
  aborted,
  searchMode,
  pruning,
  simplified
}
```

未來應補充明確停止原因，例如：

```js
stopReason: "complete" | "routes" | "time" | "nodes"
```

不得只靠 `aborted=true` 猜測是時間還是節點限制。

## 10. 修改與驗收規則

每次修改新版搜尋至少驗證：

1. Native C++ 編譯成功。
2. LTO SIMD Wasm 編譯成功。
3. 棋型與禁手自我測試成功。
4. 單組正式 ABI 測試成功。
5. 嚴格與高速剪枝確實走不同邏輯。
6. 嚴格同型表只在完整黑白盤面相同時命中。
7. 相同盤面即使查詢傳入不同 `ply` 仍應命中。
8. 黑棋或白棋多一子、少一子或位置不同時不得命中。
9. 高速模式不使用未完整搜尋的勝型子集。
10. 多組仍為先原始子集去重、再精簡。
11. 64 組或其他限制停止時仍保留已找到結果。
12. UI 顯示正確停止原因。
13. 題目產生器仍使用獨立舊版 Worker，除非該次工作明確是執行遷移。

效能修改需以相同盤面比較：

- 嚴格模式耗時與節點。
- 高速模式耗時與節點。
- 單組耗時是否退步。
- 最短手數與代表性路線是否一致。
- 精確同型表命中率與固定記憶體是否合理。

## 11. 給後續 AI 的禁止事項

處理新版 `/rapfi/` 時，不得：

- 只讀舊版 `CLAUDE.md` 就判斷現行搜尋架構。
- 把 `eval/worker.js` 當成新版主分析 Worker。
- 把舊版 `maxVCF=20` 當成新版多組上限。
- 為了修多組而替換目前最快的單組核心。
- 關閉多組強制精簡。
- 把處理順序改成先精簡再子集去重。
- 把嚴格搜尋剪枝與多組輸出子集去重混為一談。
- 在嚴格同型表中把 `ply` 或攻擊層數當成獨立 key。
- 只比較雜湊值而不驗證完整黑白 bitboard。
- 因題目預期盤面被去重，就取消一般多組的子集規則。
- 在未完成題目專用驗證前，把題目產生器重新導向 `window.engineAPI`。
- 每個節點都讀取系統時間。
- 把達到限制後的部分結果描述為完整列舉。
